## 더 빠른/더 안전한 풀이(필수)
- 아이디어: 가장 먼 집부터 역순(N-1 → 0)으로 순회하며, 처리할 배달/수거량을 변수에 누적하고, 누적량이 0을 넘는 순간 해당 위치까지 왕복 처리.
- 복잡도: O(N)
- 왜 이게 더 좋은지: 리스트를 직접 수정(`pop`, `append`)하지 않아 더 빠르고, 단일 루프로 구현이 간결해 버그 가능성이 낮습니다.

## 내가 지금 틀리기 쉬운 포인트 TOP3
- 1) 초기값/시작점/종료조건: 배달/수거가 없는 맨 뒤 집들을 미리 `pop`하는 로직은 유효하나, 전체 로직을 약간 복잡하게 만듭니다.
- 2) 인덱스/경계/오버플로/방문처리: `max(len(deliveries), len(pickups))`로 거리를 계산한 것은 한쪽 배열이 먼저 비었을 때도 잘 동작하는 정확한 방법입니다.
- 3) 자료구조/성능 함정: `pop()`과 `append()`를 반복하는 방식은 리스트 크기를 계속 변경하여 비효율적입니다. 특히 남은 양을 다시 `append`하는 구현은 가독성과 성능을 모두 해칩니다.

# 제목: 더 효율적인 단일 패스(Single-pass) 풀이
- 적용 조건: 현재 풀이의 리스트 수정(`pop`/`append`) 오버헤드를 줄이고 싶을 때.
- 핵심 아이디어: 가장 먼 집부터 역순으로 탐색하며, 처리할 배달(`d_carry`)과 수거(`p_carry`)량을 누적. `d_carry`나 `p_carry`가 0을 초과하는 순간, 해당 지점까지의 왕복이 필요하다고 판단하고 거리를 더한 뒤, 누적량에서 트럭 용량(`cap`)만큼 처리.
- 복잡도: O(N)
- 구현 스케치(Python):

def solution_alternative(cap, n, deliveries, pickups):
    answer = 0
    d_carry = 0
    p_carry = 0

    for i in range(n - 1, -1, -1):
        d_carry += deliveries[i]
        p_carry += pickups[i]

        # 현재 집에 들러야만 하는 경우 (배달/수거할 것이 남아있음)
        while d_carry > 0 or p_carry > 0:
            # 이 지점까지 와야 하므로 왕복 거리를 더함
            answer += (i + 1) * 2
            # 한 번의 왕복으로 cap만큼 처리
            d_carry -= cap
            p_carry -= cap
    
    return answer
```
