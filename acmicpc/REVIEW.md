## 한 줄 결론
- (이 문제는) **Greedy/Constructive** / 복잡도: O(N) / `minStr`을 구하는 복잡한 분기 로직이 가장 위험합니다.

## 더 빠른/더 안전한 풀이(필수)
- 아이디어: DP를 사용해 `i`개 성냥으로 만들 수 있는 최소/최대 숫자를 미리 계산합니다.
- 복잡도: O(N) (성냥 개수 N ≤ 100 가정 시, 테이블 조회는 O(1))
- 왜 이게 더 좋은지: 현재 풀이의 복잡한 `switch`문은 증명 없이 패턴에 의존해 틀리기 쉽지만, DP는 모든 경우를 체계적으로 탐색해 훨씬 안전하고 코드가 간결합니다.

## 내가 지금 틀리기 쉬운 포인트 TOP3
- 1) 최소값 로직 함정: `minStr`을 구하는 `switch`문은 특정 패턴(나머지, 몫)에 의존합니다. 이 규칙이 모든 `N`에 대해 최적임을 보장하기 어렵고, 예외 케이스 하나로 전체가 틀릴 수 있습니다.
- 2) 시작 조건(Leading Zero): `N=6`일 때 답은 '6'이지만 '0'도 가능합니다. 문제 조건에 따라 '0'이 허용되면 답이 달라질 수 있습니다. 현재 코드는 첫 자리가 0이 되는 경우를 체계적으로 고려하지 않습니다.
- 3) 경계값 처리: 성냥 개수가 적을 때(e.g., `N=10`일 때 `minStr`="22", `N=17`일 때 `minStr`="200")의 특수 케이스 처리가 복잡하게 얽혀있어 사소한 실수에도 깨지기 쉽습니다.

## 즉시 수정 액션(최대 4개)
- `minStr` 계산 로직을 아래와 같이 더 안전하고 일반적인 DP 방식으로 교체하는 것을 강력히 권장합니다.
- `maxStr` 로직은 간단하지만, "가장 성냥을 적게 쓰는 '1'을 최대한 사용"하는 그리디 전략임을 주석으로 명시하면 좋습니다.
- 현재 `minStr` 로직을 유지한다면, 각 `case`가 어떤 원리로 최적해를 구성하는지 논리를 주석으로 설명해야 합니다.
- `switch`문의 `default:`는 아무 동작도 하지 않으므로 제거해도 무방합니다.

- 핵심 아이디어: 두 개의 DP 테이블을 사용합니다. `dp_any[i]`는 0으로 시작 가능한 i개 성냥의 최소 숫자, `dp_min[i]`는 1-9로 시작하는 최소 숫자입니다. `dp_any`를 먼저 채우고, 이를 이용해 `dp_min`을 계산하면 모든 경우를 안전하게 처리할 수 있습니다.
- 복잡도: O(N) (N=최대 성냥 개수. 테이블 사전 계산 후 조회). 문자열 연산 포함 시 `O(N^2)`.

string dp_any[101]; // 0~9로 시작
int cost[] = {6, 2, 5, 5, 4, 5, 6, 3, 7, 6};

void precompute() {
    // 1. dp_any 테이블 채우기 (0으로 시작 가능)

    // base cases: dp_any[2]="1", ..., dp_any[6]="0", ...

    // dp_min[i] = min(dp_any[i-cost[d]] + d) for d=1..9
    // base cases: dp_min[2]="1", ..., dp_min[6]="6", ...
}
