안녕하세요! 백준 11003번 '최솟값 찾기' 문제 코드 리뷰를 시작하겠습니다. 전반적으로 슬라이딩 윈도우 최솟값을 찾는 매우 효율적인 표준 풀이를 잘 구현하셨습니다.

## 1) 요약
슬라이딩 윈도우 내의 최솟값을 효율적으로 찾기 위해 **모노토닉 덱(Monotonic Deque)** 자료구조를 활용한 풀이입니다. 덱에는 현재 윈도우에 포함될 수 있는 값들의 {인덱스, 값}을 저장하되, 항상 값 기준으로 오름차순을 유지합니다. 이 덕분에 덱의 맨 앞(front)에는 항상 현재 윈도우의 최솟값이 위치하게 되어 매번 O(1)에 최솟값을 조회할 수 있습니다.

- **시간 복잡도**: O(N)
- **공간 복잡도**: O(L)

## 2) 시간 / 공간 복잡도 점검
문제의 제약 조건은 N이 최대 5,000,000으로 매우 큽니다. 따라서 O(N log N)이나 O(N*L) 같은 알고리즘은 시간 초과가 발생합니다.

- **시간 복잡도**: O(N)
  - 각 숫자는 덱에 정확히 한 번 삽입(`push_back`)되고 최대 한 번 제거(`pop_front` 또는 `pop_back`)됩니다.
  - `for` 루프가 N번 반복되고, 내부의 `while` 루프는 전체 실행 동안 각 원소를 최대 한 번만 제거하므로 총 시간 복잡도는 O(N)이 됩니다.
  - N=5,000,000일 때 O(N) 알고리즘은 충분히 시간 내에 통과할 수 있습니다. **안전합니다.**

- **공간 복잡도**: O(L)
  - 덱에는 최대 L개의 원소가 저장될 수 있습니다. (윈도우 크기만큼)
  - L이 N과 같을 수 있으므로 최악의 경우 O(N)의 공간을 사용할 수 있으며, `pair<int, int>`가 약 8바이트이므로 5,000,000 * 8 = 40MB 정도의 메모리를 사용합니다. 이는 문제의 메모리 제한(보통 256MB ~ 512MB) 내에 있으므로 **안전합니다.**

## 3) 구현/버그 리스크
현재 코드는 정답이며 잘 동작하지만, 몇 가지 사소한 위험 요소를 짚어보겠습니다.

- **윈도우 범위 관리**:
  ```cpp
  if (ptr >= l and mydeque.front().first == ptr - l)
      mydeque.pop_front();
  ```
  이 코드는 현재 윈도우 `[ptr - l + 1, ptr]`에서 벗어난 원소(인덱스가 `ptr - l`인 원소)를 제거하는 역할을 합니다. 현재 로직은 정확히 동작합니다. 다만, `mydeque.front().first <= ptr - l` 과 같이 부등호로 검사하는 것이 조금 더 일반적이고 안전한 패턴입니다. 덱의 맨 앞에 여러 개의 오래된 원소가 쌓일 가능성은 없지만, 일반적인 슬라이딩 윈도우 문제에서는 부등호 체크가 더 견고할 수 있습니다.

- **가독성이 약간 떨어지는 `while` 문**:
  ```cpp
  while (!mydeque.empty())
      if (mydeque.back().second >= tmp)
          mydeque.pop_back();
      else break;
  ```
  `while-if-else-break` 구조는 동작은 하지만, 조건을 `while` 문 자체에 포함시키는 것이 더 직관적입니다.
  ```cpp
  // 제안
  while (!mydeque.empty() && mydeque.back().second >= tmp) {
      mydeque.pop_back();
  }
  ```
  이렇게 작성하면 "덱이 비어있지 않고, 마지막 값이 새 값보다 크거나 같은 동안 계속 제거한다"는 의도가 명확하게 드러납니다.

## 4) 엣지 케이스 체크리스트
- **최소 입력**: `N=1, L=1` -> 정상 동작
- **최대 입력**: `N=5,000,000, L=5,000,000` -> O(N) 풀이라서 문제 없음
- **L = 1**: 윈도우 크기가 1인 경우. 덱에는 항상 원소가 1개만 들어가고, 매번 자기 자신이 최솟값으로 출력됨. 정상 동작
- **L = N**: 전체 구간에서의 최솟값이 누적되어 출력됨. 정상 동작
- **중복 값**: 입력 `A`에 중복된 값이 많을 경우 (`{5, 5, 5, 5, 5}`). `mydeque.back().second >= tmp` 조건에 의해 이전 값들이 제거되고 가장 오른쪽에 있는 값만 남게 됩니다. 이는 최솟값을 찾는 데 문제가 없습니다.
- **정렬된 값**:
    - **오름차순** (`{1, 2, 3, 4, 5}`): 덱에서 `pop_back`이 거의 일어나지 않고, 덱의 크기는 L에 가까워짐. 정상 동작.
    - **내림차순** (`{5, 4, 3, 2, 1}`): 매번 새로운 값이 들어올 때마다 덱이 비워지고 새 값만 남게 됨. 덱의 크기는 항상 1. 정상 동작.

## 5) 리팩토링/가독성 개선 (선택)
- **변수명**: `ptr`은 포인터를 연상시키므로 `i` 또는 `currentIndex`와 같이 루프 인덱스를 나타내는 관례적인 변수명을 사용하면 가독성이 더 좋아집니다.
- **타입 별칭(Type Alias)**: `typedef`는 C 스타일입니다. C++11 이상에서는 `using`을 사용하는 것이 권장됩니다.
  ```cpp
  // using Node = pair<int, int>; // C++11 스타일
  ```
- **입출력 최적화**: `ios::sync_with_stdio(false); cin.tie(NULL);`은 N이 클 때 필수적이며, 잘 적용하셨습니다. `cout.tie(NULL)`은 `cin`과 `cout`을 번갈아 사용하지 않는다면 필수는 아니지만, 습관적으로 넣어두는 것은 좋습니다.
