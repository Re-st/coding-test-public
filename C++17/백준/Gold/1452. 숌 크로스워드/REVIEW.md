## 더 빠른/더 안전한 풀이(필수)
- 아이디어: 5중 루프와 전처리 테이블을 활용해 복잡도를 한 단계 낮춥니다.
- 복잡도: O(L^5)
- 왜 이게 더 좋은지: 마지막 단어를 배치하는 2중 루프를 O(1) 룩업으로 대체하여, 더 빠르고(L=15일 때 약 15배) 안전하며 논리가 명확해집니다.
```cpp
// O(L^5) 풀이의 핵심 아이디어 스니펫
// H1, H2, V1, V2 역할이 정해졌다고 가정
// 1. V2에 대해 (문자1, 문자2, 거리) -> 개수 맵을 전처리 O(L^2)
map<tuple<char, char, int>, int> v2_counts;
// ... 전처리 로직 ...

// 2. H1, H2, V1의 교차점 루프 O(L^5)
for (int i, j ...) { // H1-V1 교차점
  for (int p, q ...) { // H2-V1 교차점
    // ... dist_y 계산 ...
    for (int k ...) { // H1-V2 교차점
      // ... dist_x 계산 ...
      int s = p +/- dist_x; // H2-V2 교차점 x좌표 유추
      count += v2_counts[{H1[k], H2[s], dist_y}]; // O(1) 룩업
}}}}
```

## 내가 지금 틀리기 쉬운 포인트 TOP3
- 1) 조합/순열: 4개 단어를 H1,H2,V1,V2 역할에 배정하는 `P(4,4)=24`가지 경우를 모두 고려해야 합니다. 현재 코드는 `word[0]`를 H1으로 고정하고 일부 순열만 계산 후 `*2`를 하는데, 단어 길이가 다르면 이 대칭성이 성립하지 않을 수 있어 위험합니다.
- 2) 인덱스/좌표계: `y3`, `y4`는 각각 V1, V2의 시작 y좌표인데, `w3[-y3]`처럼 음수 연산으로 인덱스를 계산하는 방식은 혼란을 유발하고 버그를 만들기 쉽습니다.
- 3) 성능 함정: 6중첩 루프는 최악의 경우(모든 문자가 'a' 등) 시간 초과 위험이 있습니다. 단어 길이가 15일 때 `15^6`은 약 1100만이며, 24개 순열에 대해 수행하면 수억 단위 연산이 될 수 있어 현재 통과는 운이 좋은 케이스일 수 있습니다.
