## 한 줄 결론
- **백트래킹 완전탐색** / 복잡도: O(4^5 * N^2) / DFS 재귀 호출 시 보드 방향이 틀어지는 논리 오류가 가장 위험합니다.

## 더 빠른/더 안전한 풀이(필수)
- 아이디어: `transpose`/`flip` 대신, 네 방향(상/하/좌/우) 이동 함수를 명시적으로 각각 구현합니다.
- 복잡도: O(4^5 * N^2) (동일)
- 왜 이게 더 좋은지: 코드 가독성이 높고, 현재 코드의 치명적 버그인 ‘보드 방향성 오류’를 원천적으로 방지할 수 있습니다.

## 내가 지금 틀리기 쉬운 포인트 TOP3
- 1) **재귀 호출 로직**: `transpose` 이후 원상 복구를 안 해 보드 방향이 틀어집니다. 예를 들어 ‘위’로 이동(`transpose` -> `left`) 후 다음 `dfs`에서 ‘왼쪽’ 이동을 하면, 실제로는 원본 보드의 ‘위’ 방향으로 이동하는 것과 같게 되어 오작동합니다.
- 2) **비효율적 복사**: `dfs`와 대부분의 이동 함수가 `Board`를 값으로 전달받아, 매 재귀 호출마다 불필요한 O(N^2) 복사 비용이 발생합니다.
- 3) **이동 및 병합 구현**: ‘한 번의 이동에 한 번만 합쳐지는 규칙’과 ‘이동 방향 쪽 블록부터 합쳐지는 규칙’은 구현이 까다롭습니다. 현재 `left` 함수는 이 규칙을 잘 구현했지만, 언제나 실수하기 쉬운 부분입니다.

## 즉시 수정 액션(최대 4개)
- 1. **(치명적)** `dfs` 내에서 이동 후 보드 방향을 원래대로 복구하세요.
   ```cpp
   // 예시: '위'로 이동 후 재귀 호출
   Board temp = transpose(board);
   temp = left(temp);
   dfs(remSteps - 1, transpose(temp)); // transpose를 한번 더 하여 원상 복구
   ```
- 2. `dfs`와 헬퍼 함수들의 파라미터를 `const Board& board`로 변경하여 불필요한 보드 복사를 방지하세요.
- 3. `dfs` 내 불필요한 `upBoard.clear()`, `downBoard.clear()` 코드를 제거하세요 (지역 변수라 자동 소멸).
- 4. `left` 함수 내에서 새 `res_row`를 만들지 않고, 큐(queue)나 `O(1)` erase/insert가 가능한 `deque`를 사용하면 더 효율적인 인플레이스(in-place) 수정이 가능합니다.
