## 한 줄 결론
- **DP와 역추적** / 복잡도: O(N*M) / 역추적 시 `if-else` 조건 우선순위가 틀리면 오답 LCS가 나올 수 있음

## 더 빠른/더 안전한 풀이(필수)
- 현 접근이 거의 최선. 문제에서 LCS 문자열 자체를 요구하므로, DP 테이블 전체를 저장 후 역추적하는 O(N*M) 풀이가 가장 표준적이고 확실합니다. (N, M <= 1000)

## 내가 지금 틀리기 쉬운 포인트 TOP3
- 1) 초기값/시작점/종료조건: 역추적 시작은 `(n1, n2)`, 종료는 `i > 0 && j > 0`입니다. `dp[i][j] > 0`으로 종료 조건을 잡으면 LCS가 중간에 끊길 수 있습니다.
- 2) 인덱스/경계/오버플로/방문처리: DP 테이블은 1-based 인덱스(`i`, `j`), 문자열은 0-based 인덱스(`i-1`, `j-1`)를 사용합니다. 이 변환에서 `off-by-one` 에러가 발생하기 가장 쉽습니다.
- 3) 자료구조/성능 함정: 문자열을 뒤에서부터 `res += ...`로 만들고 마지막에 `reverse`하는 것은 효율적입니다. 매번 앞에 붙이는 `res = s1[i-1] + res`는 문자열 길이에 비례해 느려집니다.

## 즉시 수정 액션(최대 4개)
- 전역 `dp` 배열 대신 `main` 내 `vector<vector<int>>`를 선언하면 함수화 등 코드 확장에 유리하고 스택 오버플로 위험이 없습니다.
- 역추적 `else` 블록은 `s1[i-1] == s2[j-1]`인 경우인데, 이 조건이 암묵적입니다. `if (s1[i-1] == s2[j-1])`으로 조건을 명시하면 가독성이 높아집니다.
  ```cpp
  // 수정 제안 (가독성)
  if (s1[i - 1] == s2[j - 1]) {
      res += s1[i - 1];
      i--; j--;
  } else if (dp[i - 1][j] > dp[i][j - 1]) {
      i--;
  } else {
      j--;
  }
  ```
