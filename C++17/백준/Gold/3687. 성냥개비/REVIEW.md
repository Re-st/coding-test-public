## 한 줄 결론
- **[Greedy/DP]** / 복잡도: O(N) per test case / 하드코딩된 최소값 생성 로직의 정확성

## 더 빠른/더 안전한 풀이(필수)
- 아이디어: DP를 사용해 성냥개비 `i`개로 만들 수 있는 최소 숫자 `dp[i]`를 미리 계산합니다.
- 복잡도: 전처리 O(N), 쿼리 O(1)
- 왜 이게 더 좋은지: 하드코딩된 규칙보다 훨씬 체계적이고, 버그 발생 가능성이 낮으며, 로직을 증명하기 쉽습니다.

## 내가 지금 틀리기 쉬운 포인트 TOP3
- 1) 초기값/경계조건: `n=2~7` 등 작은 수에 대한 최소값은 규칙의 예외가 많아(예: n=6일 때 "6"이 "111"보다 작음) 따로 처리해야 합니다. 현재 코드는 `switch`문으로 이를 해결하려 하지만 복잡합니다.
- 2) 로직의 함정: 최소 숫자는 (1)자릿수가 가장 적고, (2)사전 순으로 가장 작은 수입니다. 이 두 조건을 동시에 만족시키는 규칙을 찾는 것은 매우 까다로우며, 현재 `switch`문은 이를 암시적으로 구현하여 검증이 어렵습니다.
- 3) 구현 복잡성: `n%7` 값과 전체 자릿수(`digit`)에 따라 분기하는 로직이 매우 복잡합니다. 예를 들어 `n%7 == 3`일 때 `n=3, 10, 17...`에 대한 처리가 모두 다릅니다. 이는 실수를 유발하기 쉽습니다.

## 즉시 수정 액션(최대 4개)
- `minStr`을 생성하는 `switch`문의 각 `case`에 어떤 `n` 값을 처리하는지 주석을 추가하여 의도를 명확히 하는 것을 권장합니다.
- `digit` 변수가 접두사 생성 후 '8'의 개수를 세는 카운터로 역할이 바뀌므로, `num_eights` 등으로 변수명을 바꿔 가독성을 높일 수 있습니다.
- `case 3`의 중첩 `if`문처럼 특정 `n`값(3, 10)을 하드코딩하는 부분은 잠재적 오류 지점이므로, DP로 전환하는 것을 강력히 추천합니다.

```REVIEW_EXAMPLE
# 제목: DP를 이용한 안정적인 최소값 계산
- 적용 조건: N의 범위가 작아(<=100) 전처리가 가능한 모든 경우
- 핵심 아이디어: `dp[i]`를 `i`개 성냥개비로 만들 수 있는 가장 작은 수(문자열)로 정의합니다. `dp[i]`는 `dp[i-k]`(k는 각 숫자를 만드는 성냥개비 수)에 해당 숫자를 덧붙인 값들 중 가장 작은 값으로 갱신합니다.
- 복잡도: O(N * |사용 가능 숫자|)
- 구현 스케치(C++):
string dp[101];

// 문자열 비교 함수 (길이가 짧거나, 길이가 같으면 사전순)
string min_str(string a, string b) {
    if (a.length() != b.length()) return a.length() < b.length() ? a : b;
    return min(a, b);
}

void precompute() {
    int cost[] = {6, 2, 5, 5, 4, 5, 6, 3, 7}; // digits 0-8
    fill(dp, dp + 101, string(51, '9')); // 매우 큰 값으로 초기화

    dp[2]="1"; dp[3]="7"; dp[4]="4"; dp[5]="2"; dp[6]="6"; dp[7]="8";

    for (int i = 8; i <= 100; ++i) {
        for (int j = 0; j <= 8; ++j) { // 9는 6과 비용이 같으므로 생략
             if (i - cost[j] >= 2) {
                 dp[i] = min_str(dp[i], dp[i - cost[j]] + to_string(j));
             }
        }
    }
}
```
