## 한 줄 결론

## 더 빠른/더 안전한 풀이(필수)
- 아이디어: `bool visited[N]` 배열로 경사로 설치 여부를 기록하며, 높이 변화 시 L칸의 평평함/겹침/범위를 직접 확인.
- 복잡도: O(N^2) (동일)
- 왜 이게 더 좋은지: 현재 풀이는 핵심 로직이 누락되었고 상태 관리가 복잡하여 수정이 어렵습니다. 제안하는 방식은 조건들을 명시적으로 검사하여 훨씬 안전하고 명확합니다.

## 내가 지금 틀리기 쉬운 포인트 TOP3
- 1) **핵심 조건 누락**: 경사로를 놓을 바닥(길이 L)의 모든 칸 높이가 같은지 확인하는 로직이 없습니다. 이 때문에 높이가 제멋대로인 길도 통과시킬 수 있습니다.
- 2) **경계 검사 오류**: 내리막 경사로(`case -1`) 설치 시, 경사로 끝(`j+L-1`)이 지도 밖으로 나가는지 확인하지 않아 런타임 에러나 오작동을 일으킬 수 있습니다.
- 3) **로직의 비약**: `lastup`, `lastdown`만으로 경사로 설치 가능 여부를 판단하려 합니다. 이는 경사로 사이의 땅이 평평한지를 보장하지 않아, `3 3 2 1 1 2 2`(N=7, L=2)와 같은 길을 잘못 통과시킬 수 있습니다.

# 제목: 더 안전한 풀이 스케치
- 적용 조건: 현재 풀이의 복잡한 상태 관리가 버그를 유발할 때
- 핵심 아이디어: `bool` 배열로 경사로 설치 위치를 명시적으로 추적. 높이 변화 지점에서 경사로 설치 조건을(범위, 평평함, 미설치) 하나씩 직접 검사.
- 복잡도: O(N^2)
- 구현 스케치(C++):
bool check(const vector<short>& path) {
    vector<bool> placed(n, false);
    for (int i = 0; i < n - 1; ++i) {
        int diff = path[i] - path[i+1];

        if (abs(diff) > 1) return false;
        if (diff == 0) continue;

        if (diff == 1) { // 내리막: i -> i+1
            for (int j = 1; j <= l; ++j) {
                if (i + j >= n || path[i+1] != path[i+j] || placed[i+j]) return false;
            }
            for (int j = 1; j <= l; ++j) placed[i+j] = true;
        } else { // 오르막: i -> i+1 (diff == -1)
            for (int j = 0; j < l; ++j) {
                if (i - j < 0 || path[i] != path[i-j] || placed[i-j]) return false;
            }
            for (int j = 0; j < l; ++j) placed[i-j] = true;
        }
    }
    return true;
}
