## 한 줄 결론
- **슬라이딩 윈도우** / 복잡도: O(S) / `dnaint` 벡터 사용으로 인한 O(S)의 불필요한 추가 메모리

## 더 빠른/더 안전한 풀이(필수)
- 현 접근이 거의 최선입니다. 슬라이딩 윈도우는 O(S)로 이 문제의 최적 해법입니다.
- 다만, `dnaint` 벡터를 생성하지 않고 원본 문자열 `dna`를 직접 순회하면 O(S)의 추가 메모리 사용을 피할 수 있어 더 효율적이고 안전합니다.

## 내가 지금 틀리기 쉬운 포인트 TOP3
- 1) 초기값/시작점/종료조건: 첫 윈도우(`0` ~ `p-1`)를 구성하는 로직이 메인 루프와 섞여 있습니다. 첫 윈도우를 별도로 계산하고, 그 후 루프를 `p`부터 시작하면 코드가 더 명확해집니다.
- 2) 인덱스/경계/오버플로/방문처리: 변수 `s`와 `p`가 문제 설명의 `|S|`, `|P|`와 반대로 사용되어 혼동을 줄 수 있습니다. (`cin >> s >> p`이므로 코드 내에서는 일관되지만, 문제와 비교할 때 헷갈립니다.)
- 3) 자료구조/성능 함정: `vector<int> dnaint`는 불필요한 O(S) 공간을 차지합니다. S가 1,000,000까지 커질 수 있으므로 메모리 초과 위험을 줄이는 것이 좋습니다.

## 즉시 수정 액션(최대 4개)
- 1. `dnaint` 벡터를 제거하고, `dna` 문자열을 직접 순회하며 `switch`문으로 인덱스를 얻으세요.
- 2. (선택) 첫 윈도우(`0` ~ `p-1`) 상태를 먼저 계산 후, `p`부터 슬라이딩하는 구조로 변경하여 초기화와 루프 로직을 분리하세요.
- 3. (선택) `if (acgt[0] >= ...)` 조건문을 `check()` 같은 함수로 분리하면 가독성이 향상됩니다.
- 4. (선택) 코드 가독성을 위해 변수 `s`와 `p`를 `S_len`, `P_len` 등으로 변경하는 것을 고려하세요.

```REVIEW_EXAMPLE
# 제목: 메모리 최적화 및 구조 개선 풀이 스케치
- 적용 조건: 현재 풀이와 시간 복잡도는 같지만, O(S) 추가 메모리를 제거하고 싶을 때.
- 핵심 아이디어: 문자열을 정수 배열로 변환하지 않고 직접 사용합니다. 첫 윈도우를 먼저 구성하고, 이후 한 칸씩 슬라이딩하며 개수를 갱신합니다.
- 복잡도: O(S) 시간, O(1) 공간 (문자열 저장 공간 제외)
- 구현 스케치(C++):
int check() {
    // 현재 윈도우의 acgt 배열이 acgtreq 조건을 만족하는지 검사
    if (acgt[0] >= acgtreq[0] && acgt[1] >= acgtreq[1] && acgt[2] >= acgtreq[2] && acgt[3] >= acgtreq[3]) {
        return 1;
    }
    return 0;
}

// ... main 함수 내 ...
// 1. 첫 윈도우(0 ~ p-1) 구성
for (int i = 0; i < p; ++i) {
    // dna[i]에 따라 acgt 배열 갱신
}
int result = check();

// 2. 슬라이딩 (p ~ s-1)
for (int i = p; i < s; ++i) {
    // 이전 문자(dna[i-p]) 제거, 새 문자(dna[i]) 추가
    result += check();
}
cout << result;
```
