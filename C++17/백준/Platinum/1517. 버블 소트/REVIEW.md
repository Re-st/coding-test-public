## 한 줄 결론
- (이 문제는) **[병합 정렬 기반 Inversion Counting]** / 복잡도: O(N log N) / swap 횟수 누적 변수 오버플로

## 더 빠른/더 안전한 풀이(필수)
- 현 접근이 거의 최선입니다. O(N log N)은 비교 기반 문제에서 이론적 최적에 가깝고, 병합 정렬을 이용한 Inversion Counting은 가장 표준적이고 직관적인 풀이입니다.

## 내가 지금 틀리기 쉬운 포인트 TOP3
- 1) 초기값/시작점/종료조건: 재귀의 베이스 케이스 `if (s >= e) return 0;`가 중요합니다. `s > e`로 잘못 설정하면 크기 1인 배열에서 무한 재귀가 발생할 수 있습니다.
- 2) 인덱스/경계/오버플로: `acc += mid + 1 - p1;`가 Inversion을 세는 핵심입니다. `+1`을 빼먹거나, `acc` 변수를 `long long`으로 선언하지 않으면 오답/오버플로가 발생합니다. (N=500,000일 때 최대 swap 횟수는 약 1.25 * 10^11)
- 3) 자료구조/성능 함정: 재귀 호출마다 `vector`를 생성/복사하면 성능 저하 및 스택 오버플로 위험이 큽니다. 전역 `tmp` 배열을 재활용하는 현재 방식이 효율적입니다.

## 즉시 수정 액션(최대 4개)
- 현재 코드는 매우 효율적이고 정확하여 즉시 수정할 부분은 없습니다.
- (참고) `mid = (e + s) / 2;` 대신 `mid = s + (e - s) / 2;`를 사용하면, `s+e`가 자료형의 최댓값을 넘는 엣지 케이스에서 오버플로를 방지하는 더 안전한 습관입니다. (이 문제에선 불필요)
